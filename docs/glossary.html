Pipeline concepts

Inpaintingconcept

Regenerating a specific region of an image based on a mask + prompt, leaving the rest untouched. Used here for detail recovery (logos, stitching, textures) — not full-scene generation.

Stitchingconcept

Compositing the inpainted region back into the source image with feathered edges so the seam isn't visible.

Avatarconcept

A brand-specific synthetic model identity. Includes a pose library + lighting library so the same avatar can render across angles without drifting.

Try-onconcept

Rendering a garment on an avatar across multiple angles. Implemented as a collage technique that preserves avatar likeness when generating new views.

Crop shotsconcept

Tight image crops of product detail. Today the pipeline produces full / upper / lower body only; tighter crops need dev work.

Spreadsheet logicconcept

The driver for batch combinations: which avatar × which garment × which pose. Pipeline reads this and queues API calls.

Naming conventionsconcept

Filename-driven contracts linking input garments to specific outputs. Each brand has its own — that's a normalization gap.

Project / business terms

SKUbiz

Stock-keeping unit. A single garment variant. The team handles 200–500 per brand per delivery cycle.

Tech packbiz

Apparel-industry term for the full product spec sheet (measurements, materials, construction notes). The QC team's background. Influences what they look for in review.

First-delivery approval ratingbiz

% of generated images the client accepts on first review. Team target: 90%+.

Creative directorbiz

Client-side decision-maker who owns visual direction. Team executes their vision; doesn't lead creatively.

Cloud architecture (proposed)

Mini admin interfacearch

Separate web app (NOT inside main Nines platform initially). Upload images → S3, pick workflow, fire JSON API, watch execution history. Google auth with 3-user pilot cap.

Modular pipelinesarch

4 independent module APIs: try-on, inpainting, variations, upscale. Each exposes its own JSON contract; Nines workflows later call them.

AWS on-demand server provisioningarch

Spin up a per-user-session server so multiple users don't conflict on a single-user ComfyUI box. Replaces today's local-only setup.

Content-filter workaroundarch

Regular APIs block lingerie/swimwear. Plan: open-source models on private servers + custom fine-tuning.

Canonical S3 storestrategy

All generated assets land in S3 (5thPivot side). Folder/key schema pending — must map to brand / SKU / angle / variation. Defined with Aaron + Mark before automation layer is built on top.

Batch-based trackingstrategy

Track progress at the batch level (25–50 SKUs per batch), not per-image. Decided 2026-05-28.

Pose library strategystrategy

Scrape brand sites → generate a library of pose variations → client picks from our library. Don't try to match exact reference poses.

Client selection modelstrategy

Deliver 5–10 variations per SKU; client picks one per angle; only the selected variation gets retouched. Retouching policy: never before client selection.

Workflow process + roles (Nare 2026-05-29)

Creative technologist (CT)role

Role that runs the actual generation work — prompts, Vivian/ComfyUI workflows, variant production. Hiring target: 3–4 full-time.

Project manager (PM)role

Bridge role: owns data accuracy + coordination, reviews shared client assets, prepares CT-ready summary decks. Essential — non-negotiable per Nare.

Two-tier QCprocess

Raw AI outputs: 70–80% accuracy acceptable. Final retouched assets: 100% required. Different humans / different bars.

6–7 variants per assetprocess

Generation policy with live QC during the run. Sits inside the broader 5–10 variations per SKU model from 2026-05-28.

4–5 generations per poseprocess

What the client picks from before any retouching. Client validates pose, mood, and avatar–product relationship.

5% max rejection rateprocess

Target rejection rate if the process is followed end-to-end.

Daily status cadenceprocess

Status calls 3×/week, 10 minutes during the manual run.

ct.nines.comarch

Repetitive vs customized categoriesprocess

Distinction that gates template reuse. Popular categories (tops, bottoms, sneakers) get reusable Vivian/ComfyUI templates; customized categories get one-offs per brand.

Per-brand background / lighting specsprocess

Vary by category × brand (e.g. Vans: tops ≠ bottoms; shoes distinct). Tracked separately from the global pose library.
